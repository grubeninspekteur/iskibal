= Iskibal
:toc: left
:toc-title: Contents

Iskibal is a Java business rules compiler.
It transforms business rules written in _Iskara_ (optionally embedded in AsciiDoc) into a rule model, which is then compiled to Java source code.

Iskibal is _not_ a rule engine.
It produces plain Java classes that you instantiate, populate with data, and call.

== Motivation

Many teams settle on spreadsheet-based decision tables or graphical rule notation because business stakeholders can read them easily.
The problem is that spreadsheets are nearly impossible to merge: two developers editing the same file at once will face a painful manual reconciliation.
Classic rule languages like DRL are mergeable, but lose the benefit of spreadsheets, and show too many implementation details.

Iskibal takes a different approach.
Rules are written as AsciiDoc documents with embedded Iskara source blocks and standard AsciiDoc tables for decision tables.
AsciiDoc renders to a clean, readable document for business review -- the same tables and structure they expect -- while remaining plain text that version control can diff and merge line by line.
Concurrent work on separate rules in the same file is straightforward; only edits to the same table row at the same time create conflicts, which is the same situation as any other text-based source file.

The developer side of the deal is also improved: facts, outputs, and rule conditions reference real Java types, so IDEs can check usages, and the compiled output is ordinary Java that can be debugged normally.

== How it works

Iskibal operates as a three-step pipeline:

. *Rule source parser* -- Reads `.iskara` files (pure Iskara) or `.adoc` files (AsciiDoc with embedded Iskara blocks) and produces a rule model.
. *Rule model* -- A tree of Java objects representing the parsed rule structure, independent of any particular source format or target language.
. *Rule compiler* -- Transforms the rule model into Java source code.

All three parts are loaded via the Java `ServiceLoader` mechanism and can be replaced with custom implementations.

== Quick start

=== 1. Add the Maven plugin

Add the Iskibal Maven plugin to your `pom.xml`.
It runs during the `generate-sources` phase and adds the output directory as a compile source root automatically.

[source,xml]
----
<plugin>
  <groupId>work.spell.iskibal</groupId>
  <artifactId>iskibal-maven-plugin</artifactId>
  <version>0.1.0-SNAPSHOT</version>
  <executions>
    <execution>
      <goals>
        <goal>generate</goal>
      </goals>
      <configuration>
        <packageName>com.example.rules.generated</packageName>
      </configuration>
    </execution>
  </executions>
</plugin>
----

You also need a runtime dependency on the Java compiler support library, which the generated code imports:

[source,xml]
----
<dependency>
  <groupId>work.spell.iskibal</groupId>
  <artifactId>iskibal-compiler-java</artifactId>
  <version>0.1.0-SNAPSHOT</version>
</dependency>
----

=== 2. Write rule sources

By default, the plugin scans `src/main/iskibal/` for `.iskara` and `.adoc` files.

==== Pure Iskara (`.iskara`)

[source,iskara]
----
facts {
    customer: com.example.Customer
}
outputs {
    discountPercent: BigDecimal := 0
    tier: String := "standard"
}

rule VIP "VIP customer gets 20% discount"
when
    customer.loyaltyPoints >= 1000
then
    discountPercent := 20
    tier := "vip"
end

rule LOYAL "Loyal customer gets 10% discount"
when
    customer.loyaltyPoints >= 500,
    customer.loyaltyPoints < 1000
then
    discountPercent := 10
    tier := "loyal"
end
----

==== AsciiDoc with embedded Iskara (`.adoc`)

Facts, outputs, and globals can be declared as AsciiDoc tables and definition lists.
Rules appear as source blocks with the `iskara` language tag and the `.rule` role.

[source,asciidoc]
....
= Shipping Rules

[.facts,cols="1,1,3"]
|===
| Fact name | Type | Description

| order | com.example.Order | The order being shipped.
|===

[.outputs,cols="1,1,1,3"]
|===
| Output name | Type | Initial value | Description

| freeShipping | String | "no" | Whether shipping is free.
| shippingCost | BigDecimal | 10 | Default shipping cost.
|===

.SHIP_FREE: Free shipping for large orders
[source,iskara,.rule]
----
when
    order.total >= 50
then
    freeShipping := "yes"
    shippingCost := 0
end
----
....

=== 3. Use the generated class

Each source file produces one Java class whose name is derived from the filename in PascalCase.
For example, `discount_rules.iskara` produces `DiscountRules`.

Facts are passed to the constructor.
Call `evaluate()` to fire all matching rules.
Outputs are then available via getters.

[source,java]
----
var customer = new Customer("Alice", 1500);
var rules = new DiscountRules(customer);

rules.evaluate();

BigDecimal discount = rules.getDiscountPercent(); // 20
String tier = rules.getTier();                   // "vip"
----

== Maven plugin reference

[cols="1,1,3"]
|===
| Parameter | Default | Description

| `sourceDirectory`
| `${project.basedir}/src/main/iskibal`
| Directory scanned for `.iskara` and `.adoc` rule source files.

| `outputDirectory`
| `${project.build.directory}/generated-sources/iskibal`
| Directory where generated Java source files are written.

| `packageName`
| _(unnamed package)_
| Java package name for all generated classes.

| `generateNullChecks`
| `true`
| Whether to generate null-safety guards for navigation expressions (e.g. `passport.address.street` returns `null` rather than throwing if `address` is null).

| `skip`
| `false`
| Set to `true` to skip the goal entirely. Can also be set via the `iskibal.skip` property.
|===

The goal binds to `generate-sources` by default.

== Modules

[cols="1,3"]
|===
| Artifact | Purpose

| `iskibal-rule-model`
| The rule model: sealed interfaces and records representing the parsed rule structure.

| `iskibal-parser`
| Parses `.iskara` source text into a rule model.

| `iskibal-asciidoc`
| Parses AsciiDoc documents with embedded Iskara blocks and structural fact/output tables.

| `iskibal-compiler-common`
| Shared compiler infrastructure: semantic analysis API.

| `iskibal-compiler-java`
| Compiles a rule model to Java source code. Also provides the runtime support classes referenced by generated code.

| `iskibal-maven-plugin`
| Maven plugin that drives the full parse-analyze-compile pipeline during a build.
|===

== Building from source

You need Maven 4 (the included `mvnw` wrapper will download it automatically) and a JDK 25.

[source,bash]
----
# Build everything and install to local repository
./mvnw clean install

# Build a specific module
./mvnw clean package -pl iskibal-maven-plugin -am
----

=== Performance note

AsciidoctorJ (used by `iskibal-asciidoc`) starts JRuby internally, which can be slow.
The parser sets `jruby.compile.mode=OFF` to disable JRuby JIT compilation, and `jruby.compat.version=RUBY1_9`, both improving startup time significantly.
In tests, reuse a single `AsciiDocParser` instance across all test methods rather than creating one per test.

== Name origin

Iskibal was a Babylonian king.
The name is a reference to https://openjdk.org/projects/babylon/[OpenJDK Project Babylon], which aims to bring code models and code reflection to the Java language.
